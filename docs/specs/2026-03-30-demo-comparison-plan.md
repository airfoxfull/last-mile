# Last Mile 框架 Demo 对比方案

## 统一验证场景

5 个候选框架跑同一个场景，对比谁最适合做 Last Mile 的底座。

### 场景：2 阶段流水线

```
需求: "给 README 添加中文徽章"

阶段 1: 规划
  - Agent 必须先提交执行计划（强制，不能跳过）
  - 人类审阅计划，打分 1-5
  - 满意度 >= 3 才能进入执行

阶段 2: 执行
  - Agent 按计划执行代码修改
  - 完成后提交工作报告（强制）
  - 人类审阅结果，打分 1-5

验证点:
  ✅ Agent 能不能绕过"先提计划"直接改代码？（工作流强制性）
  ✅ 人类审批能不能真的阻塞执行？（审批门控）
  ✅ 阶段之间的上下文能不能传递？（状态持久）
  ✅ 跑完后能不能提取经验？（知识沉淀潜力）
  ✅ 搭建 demo 花了多久？（开发体验）
```

## 5 个候选框架

### 1. Paperclip（已有基础）
- 方式: 修改心跳服务，加入强制步骤协议
- 验证: 能否在不引入新框架的情况下，让心跳执行变成"读交接→提计划→等审批→执行→写报告"的强制流程
- 优势: 已经跑通了 Agent 执行，改造成本最低
- 风险: 心跳服务 2800 行，改动可能影响稳定性

### 2. Symphony
- 方式: fork 后扩展，加入多阶段和人类审批
- 验证: 能否从"单 Agent 单任务"扩展为"多阶段链式流转"
- 优势: 编码执行最强，Elixir 容错
- 风险: Elixir 生态，团队可能不熟悉

### 3. Clawith
- 方式: 基于 OpenClaw 多 Agent 工作区，加入强制工作流 Hook
- 验证: Hook 系统能否强制"先规划后执行"的顺序
- 优势: 自进化 + 记忆持久 + TypeScript
- 风险: Hook 是"约束"不是"强制"，Agent 可能绕过

### 4. Hatchet
- 方式: 定义 DAG 工作流，每个步骤调用 AI Agent
- 验证: DAG 强制性 + 人类审批门控 + 与 Claude CLI 集成
- 优势: 工作流强制性最强
- 风险: 需要自建所有 AI 逻辑

### 5. DeerFlow 2.0
- 方式: fork 后定制，用 LangGraph 图定义流水线
- 验证: 图结构能否强制阶段顺序 + 人类审批中断
- 优势: AI 原生，已有 Agent 层级
- 风险: Python 生态，跨语言集成

## 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 工作流强制性 | 30% | Agent 能不能绕过流程？ |
| 搭建速度 | 20% | 从零到跑通 demo 要多久？ |
| 人类审批体验 | 15% | 审批是否真的阻塞？体验如何？ |
| 扩展潜力 | 15% | 能否扩展到完整的链式流转 + 预算激励？ |
| 技术栈匹配 | 10% | 和现有代码（TypeScript）的兼容性 |
| 社区生态 | 10% | 文档、社区、长期维护 |
