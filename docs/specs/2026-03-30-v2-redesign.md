# Last Mile v2 — 深度重构设计

> 从简陋 PoC 到成熟 AI-native 系统

## 核心理念变化

| v1 (当前) | v2 (目标) |
|-----------|-----------|
| Agent 是数据库记录 | Agent 是有自我意识的实体 |
| 协作靠 JSON 字段 | 协作靠版本化文档 |
| 总控是一次性函数调用 | 总控是持久活跃的 Agent |
| 没有 Skill | 每个角色有专属 Skill |
| 状态在上下文里 | 状态在文档里，跨会话持久 |
| 人看表单 | 人看 Agent 的思考和文档 |

## 1. 文档驱动协作（借鉴 Paperclip）

### 1.1 文档模型

新增 `documents` 和 `document_revisions` 表：

```sql
CREATE TABLE documents (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  doc_type TEXT NOT NULL,  -- 'plan'|'handoff'|'knowledge'|'instructions'|'report'|'review'
  owner_agent_id TEXT REFERENCES agent_accounts(id),
  linked_run_id TEXT REFERENCES pipeline_runs(id),
  linked_stage_run_id TEXT REFERENCES stage_runs(id),
  latest_body TEXT NOT NULL DEFAULT '',
  latest_revision INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE document_revisions (
  id TEXT PRIMARY KEY,
  document_id TEXT NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  revision_number INTEGER NOT NULL,
  body TEXT NOT NULL,
  change_summary TEXT,
  author_agent_id TEXT REFERENCES agent_accounts(id),
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(document_id, revision_number)
);
```

### 1.2 文档类型与用途

| 类型 | 谁写 | 谁读 | 用途 |
|------|------|------|------|
| `instructions` | 系统/人类 | Agent 自己 | Agent 的角色定义、约束、技能 |
| `plan` | 总控/阶段 Agent | 下游 Agent、人类 | 需求拆解、阶段计划 |
| `handoff` | 上游 Agent | 下游 Agent | 阶段交接：目标、上下文、约束 |
| `report` | 阶段 Agent | 上游 Agent、人类 | 工作报告：做了什么、为什么、结果 |
| `review` | 验收 Agent | 人类、总控 | 验收报告：通过/失败、发现 |
| `knowledge` | 总控 | 所有 Agent | 公司/项目知识、历史经验 |

### 1.3 协作流程（文档驱动）

```
需求进入 →
  总控写 [plan] 文档（需求分析、阶段目标、验收标准）→
  总控为每个阶段写 [handoff] 文档（目标、上下文、约束、预算）→

  阶段 Agent 读 [handoff] + 自己的 [instructions] →
  阶段 Agent 写 [plan] 文档（自己的执行计划）→
  人类审阅 plan，给满意度 →
  阶段 Agent 执行，写 [report] 文档 →
  阶段 Agent 为下游写 [handoff] 文档 →
  人类审阅 report，给满意度 →

  最后阶段完成 →
  验收 Agent 读所有 [report] + 原始需求 →
  验收 Agent 写 [review] 文档 →

  总控读所有文档 →
  总控写 [knowledge] 文档（经验沉淀）
```

## 2. Agent 自我意识（借鉴 Paperclip Agent Instructions）

### 2.1 Agent Instructions 模板

每个 Agent 有一份 `instructions` 文档，内容结构：

```markdown
# {agent.name} — {agent.role}

## 身份
你是 {name}，负责 {role} 环节。你的等级是 {rank}。

## 职责
- {role-specific responsibilities}

## 约束
- 你的 token 余额是 {balance}，每个任务基础预算 {allocation}
- 满意度 >= {threshold} 会获得奖励，< {lowThreshold} 会被惩罚
- 连续成功 {promotionThreshold} 次可以晋升
- 余额低于 {demotionThreshold} 会被降级
- 你必须在预算内完成工作

## 技能
{skill list with descriptions}

## 工作流程
1. 阅读上游交接文档，理解目标和约束
2. 制定执行计划，写入 plan 文档
3. 等待人类审阅计划
4. 执行工作
5. 撰写工作报告
6. 为下游撰写交接文档
7. 等待人类审阅结果

## 沟通规范
- 所有沟通通过文档，不依赖上下文
- 文档要清晰、结构化、可追溯
- 每次修改文档要写变更摘要

## 历史表现
- 完成任务: {taskCount}
- 平均满意度: {avgSatisfaction}
- 返工率: {reworkRate}
- 连续成功: {consecutiveSuccesses}
```

### 2.2 动态 Instructions 生成

Instructions 不是静态的，而是每次任务开始时动态生成：
- 基础模板（按角色）
- + 当前预算状态
- + 相关知识资产
- + 历史表现数据
- + 本次任务的上下文

## 3. Skill 系统

### 3.1 Skill 模型

```sql
CREATE TABLE agent_skills (
  id TEXT PRIMARY KEY,
  role TEXT NOT NULL,           -- 适用角色
  name TEXT NOT NULL,
  description TEXT NOT NULL,
  skill_type TEXT NOT NULL,     -- 'technique'|'tool'|'process'|'knowledge'
  content TEXT NOT NULL,        -- Skill 的完整内容（markdown）
  effectiveness REAL DEFAULT 0, -- 历史有效性评分
  usage_count INTEGER DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 3.2 预置 Skill

**前端 (frontend):**
- 组件设计模式（原子设计、组合模式）
- 响应式布局策略
- 状态管理最佳实践
- 可访问性检查清单
- 性能优化技巧

**后端 (backend):**
- API 设计规范（RESTful、错误处理）
- 数据库查询优化
- 输入验证模式
- 缓存策略
- 并发处理

**测试 (testing):**
- TDD 工作流
- 测试金字塔策略
- Mock vs 集成测试决策
- 边界条件清单
- 回归测试策略

**安全 (security):**
- OWASP Top 10 检查清单
- 输入验证模式
- 认证/授权审查
- 敏感数据处理
- 依赖漏洞扫描

### 3.3 Skill 注入

Agent 执行时，相关 Skill 会被注入到 instructions 中：
- 按角色匹配
- 按历史有效性排序
- 高 effectiveness 的优先注入
- 注入数量受预算约束（Skill 越多，prompt 越长，消耗越大）

## 4. 总控 Agent 持久化

### 4.1 总控的持久状态

总控 Agent 维护一组持久文档：

| 文档 | 内容 | 更新频率 |
|------|------|---------|
| `company-context.md` | 公司概况、团队能力、技术栈 | 低频 |
| `project-state.md` | 当前项目状态、进行中的需求 | 每次运行后 |
| `team-assessment.md` | 各 Agent 能力评估、历史表现 | 每次运行后 |
| `knowledge-index.md` | 知识资产索引、经验总结 | 每次运行后 |
| `decision-log.md` | 重大决策记录、原因、结果 | 按需 |

### 4.2 总控的工作流程

```
新需求进入 →
  1. 读取所有持久文档（了解全局）
  2. 分析需求（消费端、复杂度、风险）
  3. 评估团队（谁适合、谁有余额、谁表现好）
  4. 制定计划（阶段、目标、预算分配、验收标准）
  5. 写 plan 文档
  6. 为每个阶段写 handoff 文档
  7. 启动流水线

运行完成后 →
  1. 读取所有 report 和 review 文档
  2. 更新 project-state.md
  3. 更新 team-assessment.md
  4. 提取知识，更新 knowledge-index.md
  5. 记录决策，更新 decision-log.md
```

## 5. 前端重构

### 5.1 核心页面重设计

**Agent 详情页（重点）：**
- 不再是简单的数字卡片
- 展示 Agent 的 instructions 文档（可编辑）
- 展示 Agent 的 Skill 列表（可管理）
- 展示 Agent 的文档历史（写过的所有文档）
- 预算事件流水 + 余额图表
- 当前任务状态

**运行详情页（重点）：**
- 不再是简单的阶段卡片
- 展示文档流：plan → handoff → report → review
- 每个文档可展开查看完整内容和修订历史
- 文档之间的引用关系可视化
- 人类在文档上直接评分和反馈

**总控面板（新增）：**
- 总控的持久文档列表（可查看/编辑）
- 团队评估视图
- 知识索引
- 决策日志

### 5.2 文档查看器组件

核心新组件——Markdown 文档查看/编辑器：
- 渲染 Markdown
- 显示修订历史（diff 视图）
- 内联评分（在文档上直接打分）
- 作者标注（哪个 Agent 写的）
